Philosophy being advanced
--------------------------
The local machine IS the test environment and it is free: a real filesystem, a
real Ollama at http://127.0.0.1:11434 with the `vibethinker` model (confirmed
live during this analysis: Ollama 0.30.8, model `vibethinker:latest` present),
and a real subprocess/Playwright path. Mocks (FakeLLMClient, FakeValidator,
FakeCli, ScriptedClient, monkeypatched urllib) should be the EXCEPTION reserved
for unreachable/error paths and determinism-critical pure units -- NOT the
default. Today the suite inverts that: the things that can actually break in
production (real generation, the two-phase decode, the `format` constraint
really constraining, end-to-end file ops under a codec, the web path) are the
LEAST exercised, because almost everything is asserted against a fake.

2. REDUNDANCY / BLAST RADIUS OF DUPLICATION (cite file::test)
================================================================================

(A) FOUR private copies of "a fake/scripted LLM client" -- the single biggest
    boilerplate duplication in the suite:
      - test_agent.py::FakeLLMClient (+ RecordingLLMClient subclass)
      - test_streaming_log.py::_ScriptedClient
      - test_validation.py::ScriptedClient
      - (and FakeValidator appears in test_agent.py while a second pass-only
        validator _PassValidator is re-declared in test_streaming_log.py)
    Same idea four times: "return scripted action JSON, optionally record the
    prompt, optionally drive on_reason/on_action." -> Collapse to ONE shared
    fixture module (tests/_fakes.py) with a single ScriptedClient and a single
    StubValidator. Removes ~60 lines of duplicated stub code and 3 redundant
    class definitions. (No test count change, but kills the boilerplate the
    task calls out.)

(B) FakeCli in test_web.py duplicates the *shape* of the real PlaywrightCli that
    test_web_live.py already drives for real. Every test_web.py assertion
    ("calls == [['goto', url]]") merely restates BrowseTool's arg-mapping
    implementation against a recorder. The same mappings are proven end-to-end
    in test_web_live.py::test_goto_reports_title / test_click_apply_reaches_the_
    form. -> Most of test_web.py is implementation-restatement (see cluster C).

(C) Codec/registry schema overlap -- the SAME JSON-schema facts asserted in
    three files:
      - test_codec.py::test_constraint_is_a_json_schema_with_maxitems
      - test_codec.py::test_constraint_unbounded_has_no_maxitems
      - test_registry_schema.py::test_action_schema_is_array_of_tool_calls
      - test_registry_schema.py::test_action_schema_max_items_caps_the_array
      - test_registry_schema.py::test_each_branch_requires_tool_and_args
      - test_toolset.py::test_build_registry_merges...  (re-asserts oneOf consts)
      - test_memory_prompt.py / test_registry_schema.py both assert "docs mention
        every tool": test_docs_mention_every_tool vs
        test_system_prompt_renders_every_tool_name_and_description
    "maxItems caps the array" is asserted in BOTH the codec test and the
    registry test against the same registry.action_schema(). -> One
    parameterized "schema shape" test owns this; codec test only asserts the
    codec *delegates* to it.

(D) Tool-result observation verbs are individually enumerated:
      - test_tools.py::test_write_append_observation_verb
      - test_tools.py::test_write_prepend_observation_verb
      - test_tools.py::test_write_then_read_observations ("you wrote the file")
    Three near-identical tests differing only in mode string and expected verb.
    -> One table-driven test: [(overwrite,"wrote"),(append,"appended"),
    (prepend,"prepended")].

(E) Write-guard pair is split where one parameterization would do:
      - test_tools.py::test_append_without_read_ok
      - test_tools.py::test_prepend_without_read_ok
    Same behavior ("non-destructive modes bypass the read-before-overwrite
    guard") asserted twice. -> One parameterized test over mode in {append,
    prepend}.

(F) FileSystem vs fs_tools double-coverage. test_filesystem.py exhaustively
    tests FileSystem.copy/move/delete/search, and test_tools.py re-tests the
    SAME operations one thin wrapper up (ManagePathTool copy/move/delete,
    SearchTool, ListDirectoryTool). The wrapper layer only adds observation
    strings + ok flags. Examples of the same behavior asserted twice:
      - test_filesystem.py::test_copy_dest_exists_raises
        ~ test_tools.py::test_manage_copy_existing_destination_errors
      - test_filesystem.py::test_search_content
        ~ test_tools.py::test_search_tool
      - test_filesystem.py::test_list_dir_marks_directories
        ~ test_tools.py::test_list_directory
    -> Keep the deep FileSystem tests (they are the real spec). Thin the tool
    tests down to ONLY what the wrapper adds (observation text + ok/err flag +
    the guards that have no FileSystem equivalent). The "does copy actually
    copy bytes" question belongs to ONE layer, not both.

(G) Trivial / low-value tests that restate the implementation:
      - test_validation.py::test_streaming_callbacks_are_forwarded_to_client
        asserts `client.last_on_reason IS on_reason` -- a pure pass-through
        identity check; restates one line of LLMValidator. Low value once a
        real-model validator test exists.
      - test_agent.py::test_static_system_used_without_provider asserts the
        system string is repeated 3x -- restates a `for` loop.
      - test_codec.py::test_turn_action_hint_mentions_json and
        test_format_instructions_* assert substring presence in prose the codec
        emits -- brittle, restate the literal string.

(H) Reporter stream is verified twice from two angles:
      - test_reporting.py::test_validator_reasoning_and_verdict_are_streamed
        (ConsoleReporter renders it)
      - test_agent.py::test_validator_stream_reaches_reporter
        (agent forwards it)
    Acceptable as a seam test, but note the overlap; the agent-side one can
    use the RecordingReporter only (it already does) and need not re-assert
    rendering.

================================================================================
3. COVERAGE GAPS -- where mocks hide real behavior
================================================================================

These are the things that would still be GREEN if the real integration broke,
because no test ever runs them for real:

  G1. THE TWO-PHASE DECODE (llm.py _reason -> _act). Only the static string
      helpers (_render_chatml, _continue_after_reasoning) are unit-tested.
      The actual phase-1 stop-at-</think>, the raw-continuation prefill, and
      phase-2 generation are NEVER executed against Ollama. If the ChatML
      template, the </think> stop, or the prefill drifted, every unit test
      still passes.

  G2. CONSTRAINED DECODING ACTUALLY CONSTRAINING. test_codec.py only checks
      that constraint() RETURNS a dict with maxItems. Nothing verifies that
      passing that schema to Ollama's `format` actually forces the model to
      emit a parseable action of the right shape, or that maxItems actually
      bounds the array. This is the highest-value untested claim in the repo
      -- the entire design rests on "format constrains output," and it is
      asserted nowhere against a real model.

  G3. END-TO-END AGENT RUN ON A REAL MODEL + REAL FILESYSTEM. RalphAgent is
      tested only with FakeLLMClient. There is no test where the real model
      reads a task, emits real tool calls, and a file actually appears on disk
      and validation passes. The agent<->codec<->client<->fs contract is
      entirely mock-verified.

  G4. THE CODEC SEAM AGAINST A REAL MODEL, PER CODEC. codec.get_codec is built
      for swappable formats (json today; tagged/xml/gbnf planned). There is no
      "every registered codec round-trips a real model's output" test, so a new
      codec could ship with instructions the model can't follow and the suite
      would not notice.

  G5. OllamaClient.generate early-stop is tested (good) but decide() -- the
      method the agent actually calls -- has NO live test at all.

  G6. THE `gbnf` / stop-string branch of DecodeConstraint is dead in tests:
      llm._act only applies json_schema; nothing exercises constraint.stop or
      documents that gbnf is intentionally ignored on the Ollama backend.

  G7. tests/integration/ IS THIN on the model side: 2 real-generation tests,
      both liveness smoke tests ("it produced text", "it stopped at ~100
      chars"). Neither touches the agent, the codec, the constraint, or tool
      execution. The web side is comparatively rich (8 real tests) -- the model
      side should reach parity.

  G8. NO MARKER / SKIP INFRASTRUCTURE. Integration files hand-roll _ollama_up()
      / _server_up() / _cli_available() and skipUnless at class scope. There is
      no pytest marker (e.g. `needs_ollama`, `needs_web`), no conftest, no
      `-m "not slow"` fast path. Adding more integration tests today means
      copy-pasting the probe boilerplate a 3rd, 4th, 5th time (see redundancy A
      for the client-stub equivalent).

WHAT STAYS A UNIT TEST (and why) -- the legitimate mock minority:
  - OllamaUnavailable on a dead port (test_model_live) -- error path, must be
    deterministic and serverless. KEEP as unit.
  - Codec/registry SCHEMA SHAPE, prompt rendering, NarrativeMemory, Settings,
    RunLogger, ConsoleReporter, FileSystem primitives -- pure, fast, real code
    already; no model needed. These are the suite's healthy core.
  - Parser ERROR cases (invalid JSON, missing tool, bad args, unknown tool,
    over-cap, fill-missing-text) -- you cannot reliably make a real model emit
    malformed output on demand; mocking the bad input is correct here.
  - The streaming-to-disk-per-turn test -- the *value* is the fs+logger wiring,
    which is already real; the scripted client is the right tool because we need
    an exact, fast turn count.

BEST PRACTICES TO ENFORCE:
  - Hermetic-but-real: every integration test uses a fresh real TemporaryDirectory
    and asserts cleanup (test_streaming_log already models this well -- copy it).
  - Determinism on a real model: temperature=0 / action_temperature=0, tiny
    token budgets, assert on ROBUST properties (file exists, contents contain a
    keyword, parse succeeds, action count bounded) NOT on exact model phrasing.
  - Fast vs slow split: default `pytest` runs only hermetic units (`-m "not
    slow"`); `pytest -m needs_ollama` runs the real-model suite locally. CI stays
    green without the model; the dev box gets full real coverage for free.
  - Skip-gracefully: a single conftest probe per resource (ollama_up/web_up),
    NOT per-file copy-paste. Down dependency -> skip with a clear reason, never
    fail.
  - One shared fakes module so the 4 duplicated client stubs become 1, and the
    line between "legitimately faked" and "should be real" is visible in one
    place.
  - Each codec added to the registry is automatically covered by the
    parametrized P0 end-to-end test -- new format, zero new test wiring.
