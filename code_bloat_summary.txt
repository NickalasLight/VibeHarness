 VIBEHARNESS — CODE SIZE & BLOAT ANALYSIS (production package: vibeharness/)
================================================================================
 Branch:   chore/code-bloat-analysis  (worktree C:\git\vh-code-bloat)
 Scope:    vibeharness/ package only. Tests analyzed at high level (separate
           effort owns test bloat).
 Method:   Measured, not estimated. Line classification via Python `tokenize`
           (comment vs. string/docstring vs. code). Growth from git history
           (`git show <rev>:<path> | wc -l`). Hotspots carry file:line evidence.

--------------------------------------------------------------------------------
 EXECUTIVE SUMMARY
--------------------------------------------------------------------------------
 The package is SMALL and, for the most part, GENUINELY LEAN: 21 files,
 ~2,299 physical lines / ~1,516 code lines. Architecture is clean (clear SOLID
 seams: LLMClient, ToolCallCodec, Toolset, Reporter, FileSystem service). This
 is NOT a bloated codebase by absolute size. The bloat that exists is local and
 cheap to fix: a duplicated streaming/error block in llm.py, one declared-but-
 never-used abstraction field (gbnf), one entirely unused method (memory.is_empty),
 a partially-dead public method (generate, used only by integration tests), and
 a high docstring-to-code ratio in a few small files (codec.py, prompt.py,
 validation.py) where prose outweighs logic.

 Estimated realistically-recoverable production LOC: ~60-90 lines (~4-6% of
 code), with zero behavior change. The biggest *quality* win (deduping the
 streaming transport) is ~25-30 LOC and also removes a real maintenance hazard
 (two copies of the Ollama error message that must be kept in sync).

 OBSERVATIONS
 - Comments are very sparse (64 lines, 2.8%). Where present they are load-bearing
   (encoding rationale, attention/primacy notes). This is healthy, not bloat.
 - Docstrings are heavy in the small "seam" files: codec.py (41 doc / 30 code),
   prompt.py (48 doc / 29 code), validation.py (36 doc / 55 code). The prose is
   high quality but the doc:code ratio is inverted — see §3F.
 - fs_tools.py is the biggest "honest" repetition: 6 Tool subclasses each with an
   __init__, a parameters property, and a run() that follows the identical
   "try -> FileSystemError -> 'you tried to X ... error' -> success sentence"
   shape. This is intentional explicitness, but it is the largest concentration
   of structural duplication in the package (see §3A).

================================================================================
 3. BLOAT HOTSPOTS — concrete, with file:line evidence
================================================================================

 -- 3A. Duplicated Ollama streaming + error block (HIGHEST-VALUE) --------------
 llm.py has TWO near-identical streamed-read loops with a BYTE-FOR-BYTE
 duplicated error handler:

   * generate()       llm.py:63-91   (one-shot raw stream, max_chars early stop)
   * _stream()        llm.py:125-152 (the transport used by _reason/_act)

 Both build a urllib Request, iterate `resp` line by line, json.loads each line,
 pull a chunk, append + fire on_token, break on done. The ONLY real differences:
 generate() supports a max_chars early-stop and reads only "response"; _stream()
 also handles the chat "message.content" shape. The except block is identical:

   llm.py:86-90   raise OllamaUnavailable(f"Could not reach Ollama at ...")
   llm.py:147-151 raise OllamaUnavailable(f"Could not reach Ollama at ...")  <- verbatim copy

 Risk: the two copies must be kept in sync; a fix to one can silently miss the other.
 Fix: extract a single `_read_stream(req, on_token, want=..., max_chars=...)` helper.
 Saves ~25-30 LOC and removes the duplicate error string. (See §4 #1.)

 -- 3B. Dead / partially-dead code --------------------------------------------
   * memory.py:17  NarrativeMemory.is_empty() — DEFINED, NEVER CALLED anywhere in
     vibeharness/ or tests/. (__len__ already covers the need.) Pure dead code.
     -> delete (~3 LOC).
   * llm.py:57-91  OllamaClient.generate() — NOT used by the agent loop at all;
     only referenced by tests/integration/test_model_live.py as a liveness probe.
     It is a 35-line public method carried solely for an integration smoke test.
     Not strictly dead, but it is product code earning its keep only via a test.
     -> keep if the liveness check matters, but it is the prime beneficiary of the
        §3A extraction (it would shrink to ~8 lines on top of the shared helper).

 -- 3C. Speculative generality — the `gbnf` constraint field -------------------
   * codec.py:42  DecodeConstraint.gbnf: str | None = None
   * codec.py:36  documented as "honoured only by a llama.cpp backend"
   No codec ever SETS gbnf (json_codec.py:34-36 sets only json_schema), and the
   LLM client never READS it (llm.py:119-120 applies json_schema; llm.py:115
   applies stop; gbnf is never referenced). It is an abstraction not yet earned by
   any use — classic speculative generality introduced "for the future".
   -> Either delete the field + its doc bullet (~3 LOC + comment) until a GBNF
      backend exists, OR leave it and acknowledge it as a known placeholder. Low
      cost either way; flagged because it is the clearest "unused abstraction".

 -- 3D. Repeated tool-result message shape (fs_tools.py / web.py) --------------
 Every filesystem tool's run() repeats the same skeleton (count = 11 "you tried
 to ..." strings in fs_tools.py, 2 in web.py):

     try:  result = self._fs.OP(...)
     except FileSystemError as e:
         return ToolResult(False, f"you tried to {verb} '{path}' but ... error: {e}.")
     return ToolResult(True, f"you {past_verb} ...")

 Evidence: fs_tools.py:53-56, 81-88, 113-116, 143-152, 177-182, 206-227.
 This is *deliberate explicitness* (each sentence is hand-tuned for the narrative
 memory), so it is NOT pure copy-paste waste — but it IS the package's largest
 structural-repetition surface (~6 classes × ~15 LOC). web.py:55-73 already shows
 the lean alternative: a single `_ACTIONS` dispatch table drives 18 browser verbs
 in ~115 LOC. The filesystem toolset could adopt the same table-driven shape.
 -> MEDIUM-confidence refactor; needs the existing tool tests green first
    (test_tools.py covers these). Potential ~30-50 LOC saved but it trades
    explicit per-tool readability for a table — judgement call, not a clear win.

 -- 3E. Codec scaffolding vs. a single implementation -------------------------
 The ToolCallCodec seam (codec.py, 98 LOC, 41 of them docstring) + the
 codecs/ package (__init__ 8 LOC, json_codec 59 LOC) exists to allow MANY wire
 formats, but only ONE (json) ships. get_codec() (codec.py:77-97) does dynamic
 importlib resolution with careful ModuleNotFoundError disambiguation — 20 LOC of
 machinery serving a single concrete codec. This is justified IF more codecs are
 imminent (the merge message and §2 suggest they are planned); if not, it is
 over-engineered relative to current use. Not "delete" material — flagged as the
 abstraction-vs-earned-use tension to watch.

 -- 3F. Docstring/comment-vs-code imbalance -----------------------------------
 Three small files carry more prose than code:
     prompt.py     48 doc / 29 code   (1.66x)
     codec.py      41 doc / 30 code   (1.37x)
     validation.py 36 doc / 55 code   (0.65x, but VALIDATOR_SYSTEM is a 19-line
                                       prompt literal — content, not bloat)
 The prose is high quality (attention/primacy reasoning in prompt.py:76-89,
 codec seam contract in codec.py:1-13). This is NOT a defect — but if line count
 is the metric, ~30 lines of explanatory docstring could be trimmed to terse
 one-liners. RECOMMEND AGAINST: this documentation is the kind worth keeping.

 -- 3G. Over-long / complex spots (mostly fine) -------------------------------
   * filesystem.py:199-249  tree() with a nested recursive walk() closure and a
     `truncated = [0]` list-as-mutable-cell — the one spot of real cyclomatic
     density. Readable but the densest function in the package. Leave unless touched.
   * cli.py:141-224 run_agent() is long (~84 LOC) but it is linear top-to-bottom
     wiring (parse -> select toolsets -> build registry -> run -> log). Low
     complexity despite length; splitting it would add indirection for little gain.
   * agent.py:103-151 RalphAgent.run() — moderate nesting in the action loop but
     each branch is short and well-commented. Acceptable.
 No function is alarmingly long or deeply nested. No god-class. No obvious
 unreachable branches besides the items in §3B/§3C.

 -- 3H. What is genuinely LEAN (call it out honestly) -------------------------
   * registry.py, toolset.py, tools.py, memory.py, config.py — tight, single-
     responsibility, no waste. config.py's comments are all rationale (GPU/ctx),
     not noise.
   * reporting.py — Null + Console pair is textbook; the streaming state flags are
     necessary, not bloat.
   * Cross-module DUPLICATION is LOW overall: aside from §3A (llm.py) and the
     stylistic §3D pattern, there is no copy-pasted helper, no repeated urllib
     logic outside llm.py, no parse/validate duplication across modules.

================================================================================
 4. RECOMMENDATIONS — prioritized & actionable
================================================================================
 (Behavior-preserving unless noted. LOC = production lines removed/avoided.)

 HIGH ----------------------------------------------------------------------------
 #1  DEDUPE the streaming transport in llm.py.   [arch smell — maintenance hazard]
     Extract one `_read_stream(req, on_token, want, max_chars=None)` and route both
     generate() (llm.py:57-91) and _stream() (llm.py:125-152) through it. Removes
     the verbatim-duplicated OllamaUnavailable block (llm.py:86-90 == :147-151).
     SAVES ~25-30 LOC. Needs tests first: tests/test_streaming_log.py,
     test_llm_helpers.py cover this path — run them green before & after.

 #2  DELETE dead code.                                            [cosmetic, safe]
     - memory.py:17 NarrativeMemory.is_empty() — unreferenced. Remove (~3 LOC).
     - Decide on llm.py generate(): keep (it's a useful liveness probe used by
       integration tests) but let it collapse onto the #1 helper (~ -25 LOC of
       its body). If the liveness test is dropped, delete it outright (~ -35 LOC).
     SAVES 3-35 LOC depending on the generate() decision.

 MEDIUM --------------------------------------------------------------------------
 #3  RESOLVE the `gbnf` speculative field.            [unused abstraction]
     codec.py:42 + its doc bullet (codec.py:36) — declared, never set, never
     applied (confirmed: only json_schema/stop are used in llm.py:115-120).
     Either delete until a llama.cpp/GBNF backend lands, or leave with an explicit
     "# not yet wired" marker. SAVES ~3 LOC + clarifies intent. Trivial risk.

 #4  TABLE-DRIVE the filesystem tools (optional).     [structural duplication]
     fs_tools.py 6 Tool classes repeat the try/except/observation shape (§3D).
     web.py:55-73 already proves the table-driven pattern works. Converting fs
     tools to a similar action/verb table COULD save ~30-50 LOC. JUDGEMENT CALL:
     it trades per-tool explicit readability for a dispatch table, and each fs
     observation sentence is individually tuned. Do ONLY if test_tools.py fully
     locks behavior first; otherwise leave as deliberate explicitness.

 LOW -----------------------------------------------------------------------------
 #5  TRIM the heaviest docstrings (cli/prompt/codec).         [cosmetic only]
     prompt.py:1-13 & :76-89 and codec.py:1-13 carry ~30 lines of explanatory
     prose. Could be condensed to terse one-liners for a pure line-count win.
     RECOMMEND AGAINST unless line count is a hard target — this is exactly the
     "why" documentation worth keeping. SAVES ~20-30 LOC but COSTS context.

 #6  WATCH the codec scaffolding (no action yet).      [earned-use tension]
     codec.py get_codec() machinery (codec.py:77-97) serves a single codec today.
     Justified only if more codecs are imminent. If the multi-codec roadmap
     stalls, revisit collapsing the dynamic-import indirection. No change now.

 -- Net realistic savings: ~60-90 production LOC (~4-6%), almost all from #1-#3,
    all behavior-preserving. The package does NOT need a large cleanup; it needs
    one targeted dedup (#1) and a little dead-code/abstraction pruning (#2,#3).

================================================================================
 BOTTOM LINE
================================================================================
 vibeharness is a well-factored ~2.3k-line package that grew by adding clean,
 single-purpose modules rather than by fattening existing ones. Real bloat is
 confined to: (1) a duplicated streaming/error block in llm.py, (2) a couple of
 dead/partially-dead members, and (3) one unused abstraction field (gbnf). Fix
 those three and the codebase is essentially as lean as its feature set allows.
 The only architecture-smell-grade item is #1 (duplicate transport); everything
 else is cosmetic or a deliberate design trade-off. Do #1 with tests in hand.
